## What CodeQL reported

This is the same pattern as [finding #4](./4.md), one line down in the same `Detect languages` step. The
`javascript-typescript` leg of the matrix was gated on a `hashFiles()` expression interpolated directly into the `run:`
shell script:

```sh
if [ -n "${{ hashFiles('package.json') }}" ]; then
  langs="$langs,{\"language\":\"javascript-typescript\",\"build-mode\":\"none\"}"
fi
```

CodeQL's `actions/code-injection` query flags the `${{ hashFiles('package.json') }}` expansion as a potential injection
sink, because a `${{ }}` expression is substituted into the script before the shell runs.

## Why it is not exploitable, and the fix

Identical reasoning to [finding #4](./4.md): `hashFiles()` returns only an empty string or a 64-character hex digest, an
alphabet with no shell metacharacters, so the expansion cannot inject code. The remediation is the same `env:`-variable
indirection — the value is now read from `PACKAGE_JSON_HASH` rather than inlined:

```yaml
- name: Resolve languages
  env:
    PACKAGE_JSON_HASH: ${{ hashFiles('package.json') }}
  run: |
    # ...
    if [ -n "$PACKAGE_JSON_HASH" ]; then
      langs="$langs,{\"language\":\"javascript-typescript\",\"build-mode\":\"none\"}"
    fi
```

See [finding #4](./4.md) for the full rationale, the `hashFiles()` output-alphabet argument, and the note on the related
`inputs.languages` hardening applied in the same step.
