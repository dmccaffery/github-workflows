# Code scanning finding #4 — Code injection via `hashFiles('go.mod')`

**Finding:**
[github.com/bitwise-media-group/github-workflows/security/code-scanning/4](https://github.com/bitwise-media-group/github-workflows/security/code-scanning/4)

| Field    | Value                                                     |
| -------- | --------------------------------------------------------- |
| Rule     | `actions/code-injection/medium` (CWE-94)                  |
| Tool     | CodeQL 2.25.6                                             |
| Severity | Warning (security-severity: medium)                       |
| Location | `.github/workflows/codeql.yaml:60`                        |
| Detected | 2026-06-20 against commit `66b14ef` (`refs/pull/8/merge`) |
| Outcome  | **Remediated**                                            |

## What CodeQL reported

The `Detect languages` step assembled the analysis matrix by interpolating a `${{ }}` expression straight into the
`run:` shell script:

```yaml
- name: Detect languages
  id: detect
  run: |
    set -eu
    langs='{"language":"actions","build-mode":"none"}'
    if [ -n "${{ hashFiles('go.mod') }}" ]; then
      langs="$langs,{\"language\":\"go\",\"build-mode\":\"autobuild\"}"
    fi
    # ...
```

CodeQL's `actions/code-injection` query flags any workflow expression expanded into a `run:` block as a potential
injection sink: the `${{ ... }}` is substituted into the script **before** the shell runs, so if the expanded value
could contain shell metacharacters an attacker could break out of the surrounding quotes and execute arbitrary code with
the job's token and secrets. The query treats `hashFiles()` as a tainted source and reports the
`${{ hashFiles('go.mod') }}` expansion on line 60. The sibling expansion `${{ hashFiles('package.json') }}` on line 63
is reported separately as [finding #5](./5.md).

## Why it is not exploitable as written

`hashFiles(path)` is a GitHub Actions built-in that returns the **SHA-256 digest** of the matched files — a fixed
64-character lowercase-hex string (`[0-9a-f]{64}`) — or the **empty string** when nothing matches. That output alphabet
contains no quotes, no `$`, no `;`, no backticks, no whitespace — nothing the shell would treat as a metacharacter. The
value is also derived from file _contents_, not copied from any attacker-supplied string. So there is no input an
attacker could craft that would make `${{ hashFiles('go.mod') }}` expand into anything other than empty-or-hex, and the
`if [ -n "…" ]` test can only ever see one of those two shapes. The finding is a true-positive _sink_ with a
non-exploitable _source_.

## Resolution

Even though the source is benign, the fix is the documented best practice for this rule and is strictly cleaner, so it
was applied rather than dismissed: the expression is bound to an intermediate `env:` variable and the script reads the
variable instead of carrying an inlined `${{ }}`.

```yaml
- name: Resolve languages
  id: detect
  env:
    GO_MOD_HASH: ${{ hashFiles('go.mod') }}
    PACKAGE_JSON_HASH: ${{ hashFiles('package.json') }}
    # LANGUAGES: ${{ inputs.languages }}  — see "Related hardening" below
  run: |
    set -eu
    # ...
    if [ -n "$GO_MOD_HASH" ]; then
      langs="$langs,{\"language\":\"go\",\"build-mode\":\"autobuild\"}"
    fi
```

A value passed through `env:` is placed into the process environment by the runner and referenced with ordinary shell
variable expansion (`"$GO_MOD_HASH"`). It is never spliced into the script text, so it cannot be parsed as code — this
is exactly the indirection CodeQL's "Recommendation" prescribes, and it breaks the taint flow the query tracks, so the
alert clears on re-analysis (unlike the reusable-workflow false positives in [#2](./2.md)/[#3](./3.md), which CodeQL
cannot statically prove and may keep open).

## Related hardening

While remediating, a third expansion in the same step was hardened the same way even though CodeQL had not (yet) flagged
it: `languages='${{ inputs.languages }}'`, added on this branch after the scan. Unlike `hashFiles()`, `inputs.languages`
is a **free-form caller-supplied string**, so a value containing a single quote could break out of the surrounding `'…'`
quoting — a genuine injection sink, and a stronger one than the two findings. It is now passed via a `LANGUAGES` env var
alongside the two hashes, closing it before the next scan would have raised it as a separate alert.
