# Code scanning finding #3 — Checkout of untrusted code in a trusted context

**Finding:**
[github.com/bitwise-media-group/github-workflows/security/code-scanning/3](https://github.com/bitwise-media-group/github-workflows/security/code-scanning/3)

| Field    | Value                                                   |
| -------- | ------------------------------------------------------- |
| Rule     | `actions/untrusted-checkout/medium` (CWE-829)           |
| Tool     | CodeQL 2.25.6                                           |
| Severity | Warning (security-severity: medium)                     |
| Location | `.github/workflows/release-go.yaml:96-102`              |
| Detected | 2026-06-15 against commit `ad8c9e3` (`refs/heads/main`) |
| Outcome  | **Remediated**                                          |

## What CodeQL reported

This is the same pattern as [finding #2](./2.md) in the sibling Node/TypeScript release pipeline. The `goreleaser` job
is privileged (`contents: write`, `id-token: write`, `attestations: write`, `artifact-metadata: write`) and checks out a
non-default ref, then runs build code over the checked-out tree:

```yaml
- name: Checkout repository at release tag
  uses: actions/checkout@... # v6.0.3
  with:
    ref: ${{ needs.release-please.outputs.tag_name }}
    fetch-depth: 0
# ...
- name: Create release (goreleaser)
  uses: goreleaser/goreleaser-action@... # v7.2.2
  with:
    args: release --clean
```

CodeQL's heuristic — privileged job + checkout of a ref it cannot prove is trusted + execution of repo code
(`goreleaser release --clean`) — flags this as a potential "pwn request".

## Why the alert fires on a reusable workflow

`release-go.yaml` is a reusable workflow (`on: workflow_call`), so CodeQL cannot see the caller's trigger and assumes
the worst case (a privileged fork-controllable event). In practice the path does not exist:

- The shipped caller (`examples/release-go.yaml`) is triggered only by `push: branches: [main]`.
- The ref is `needs.release-please.outputs.tag_name` — a `vX.Y.Z` tag computed by `release-please` from already-merged
  history, not fork-controllable input.

GitHub Actions provides no way for a reusable workflow to declare which trigger events may call it, so the safety
property is enforced at runtime rather than declared.

## Resolution

The privileged checkout-and-build job is gated on a trusted trigger context:

```yaml
goreleaser:
  needs: release-please
  if: >-
    needs.release-please.outputs.release_created && contains(fromJSON('["push","workflow_dispatch","schedule"]'),
    github.event_name)
```

Rationale, guard scope (applied to `goreleaser`, not `release-please`), why the guard is a positive allowlist rather
than a denylist of fork-triggerable events, and the note that CodeQL may still report this and can then be dismissed as
a false positive referencing the triage — all match [finding #2](./2.md); see that report for the full reasoning.
