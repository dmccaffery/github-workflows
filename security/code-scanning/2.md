# Code scanning finding #2 — Checkout of untrusted code in a trusted context

**Finding:**
[github.com/bitwise-media-group/github-workflows/security/code-scanning/2](https://github.com/bitwise-media-group/github-workflows/security/code-scanning/2)

| Field    | Value                                                   |
| -------- | ------------------------------------------------------- |
| Rule     | `actions/untrusted-checkout/medium` (CWE-829)           |
| Tool     | CodeQL 2.25.6                                           |
| Severity | Warning (security-severity: medium)                     |
| Location | `.github/workflows/release-actions.yaml:60-65`          |
| Detected | 2026-06-15 against commit `ad8c9e3` (`refs/heads/main`) |
| Outcome  | **Remediated**                                          |

## What CodeQL reported

The `publish` job is privileged (`contents: write`) and checks out a non-default ref:

```yaml
- name: Checkout repository at release tag
  uses: actions/checkout@... # v6.0.3
  with:
    ref: ${{ needs.release-please.outputs.tag_name }}
# ...
- name: Run build pipeline
  run: npm run all
```

The job then runs build code (`npm run all`) over the checked-out tree. CodeQL's heuristic — privileged job + checkout
of a ref it cannot prove is trusted + execution of repo code — flags this as a potential "pwn request" (untrusted code
running with write access and secrets).

## Why the alert fires on a reusable workflow

`release-actions.yaml` is a reusable workflow (`on: workflow_call`). When CodeQL analyses it in isolation it **cannot
see the caller's trigger**, so it assumes the worst case: that some caller might invoke it from a privileged
fork-controllable event (`pull_request_target`, `issue_comment`, `workflow_run`) where a checkout ref could be
attacker-controlled.

In practice this path does not exist:

- The shipped caller (`examples/release-actions.yaml`) is triggered only by `push: branches: [main]`.
- The ref checked out is `needs.release-please.outputs.tag_name` — a `vX.Y.Z` tag computed by `release-please` from
  already-merged commit history, not `github.event.pull_request.head.sha` or any other fork-controllable input.

So as written the finding is not exploitable. GitHub Actions provides **no way for a reusable workflow to declare which
trigger events may call it** (`workflow_call` accepts an invocation from any caller regardless of that caller's
trigger), so the safety property cannot be expressed declaratively — it can only be enforced at runtime.

## Resolution

The privileged checkout-and-build job is gated on a trusted trigger context:

```yaml
publish:
  needs: release-please
  if: >-
    needs.release-please.outputs.release_created && contains(fromJSON('["push","workflow_dispatch","schedule"]'),
    github.event_name)
```

When a reusable workflow runs, `github.event_name` reflects the caller's original triggering event. Gating on a positive
allowlist of trusted events makes the trusted-context assumption explicit and enforced: if a consumer ever wires this
workflow to `pull_request_target` (or another fork-controllable event), the privileged job is skipped rather than
running attacker-controlled build code with secrets. This is defence in depth — it does not depend on the consumer
configuring their caller correctly.

### Scope of the guard

The guard is applied only to the `publish` job, **not** to `release-please`. The `release-please` job has no
untrusted-checkout vector: it runs the `googleapis/release-please-action` and never checks out a computed ref or
executes repository code. It operates entirely through the GitHub API on already-merged history, which is why CodeQL did
not flag it. Guarding it would add no security but would over-constrain it; the privileged `publish` job's allowlist
already covers `push`, `workflow_dispatch`, and `schedule`, so on every trusted trigger both jobs run in lockstep, and
on a fork-controllable event `release-please` may run harmlessly while `publish` is skipped.

### Why an allowlist, not a denylist

The guard matches a positive list of trusted events (`push`, `workflow_dispatch`, `schedule`) rather than excluding the
known fork-controllable ones (`pull_request`, `pull_request_target`, `issue_comment`, `workflow_run`). A denylist is
fail-open: if GitHub later adds a new event type that a fork can trigger, an unmaintained denylist would silently admit
it. The allowlist is fail-closed — any event not on the list is skipped — so a newly added event type must be reviewed
and explicitly vetted before it can reach the privileged job. `push` covers the shipped examples; `workflow_dispatch`
and `schedule` let a maintainer retry or schedule a release, and both are themselves only triggerable by an actor with
write access, never by a fork.

### CodeQL may still report this

CodeQL cannot statically prove a reusable workflow's trigger, so the alert may remain open after re-analysis despite the
guard. If it does, dismiss it as a **false positive** referencing this report: the ref is a release-please-computed tag,
the shipped callers are push-triggered, and the privileged job is now gated on a trusted-event allowlist. The guard is
retained regardless of whether CodeQL goes quiet.
